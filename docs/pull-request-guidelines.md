# Руководство по Pull Request

См. также:
- [Конвенции кода](./conventions.md)
- [Предложения фич](./feature-proposals.md)
- [Git для разработчика SS14](./git-for-the-ss14-developer.md)


Спасибо за вклад в Space Station 14. При отправке pull request'ов (PR), пожалуйста, следуйте этим рекомендациям, чтобы их было проще ревьюить и мерджить.

```admonish warning
Pull request'ы, которые не соответствуют этим правилам, могут быть закрыты по усмотрению мейнтейнера.

В конечном счете именно мейнтейнеры решают, какой контент будет влит в Upstream-репозиторий.
Ваш pull request может быть отклонен или закрыт по любой причине.
```

## Перед началом

- Если вы не знакомы с Git workflow, прочитайте наш [гайд по Git](./git-for-the-ss14-developer.md) и задавайте любые вопросы в #howdoicode.

- Ознакомьтесь с [конвенциями C#](https://docs.microsoft.com/en-us/dotnet/csharp/) (если работаете с C#) и [нашими внутренними конвенциями](./conventions.md). Полезно посмотреть, как оформлены другие части кодовой базы.

## Содержимое PR

- **Делайте отдельные PR для фич, багфиксов и cleanup/refactor-изменений.** Так их проще ревьюить, меньше конфликтов, и проще откатывать, если что-то пошло не так.

    - Изменения фич и багфиксы должны быть в отдельных PR.
    - Cleanup и "refactoring", включая переименование переменных и правки отступов (например, из-за file-scoped namespace), должны быть в отдельном PR.
    - **Рефакторинги должны быть отдельным PR.** Сюда входят изменения, затрагивающие значительное число public API (поля, методы и т.д.) и требующие правок в нескольких системах. Такие изменения должны идти отдельно от контентных изменений и багфиксов.
    - Если вы переносите файл в другую папку и/или namespace, по возможности делайте это отдельным коммитом, чтобы было проще отличить реальное изменение содержимого от простого переноса. В случае невозможности делать всё одним коммитом из-за длительной работы используйте Squash в Git.
    - Изменения карт должны идти отдельным PR для каждой измененной карты. Даже если изменение совсем небольшое, с обязательным приложением медиа.

- **Не объединяйте несколько несвязанных изменений в одном PR.** Например, не добавляйте разнородные мелочи: не стоит одновременно с PR на новую пушку менять сопротивление проникающему урона у скафандра.

    - Старайтесь разбивать PR на более мелкие там, где это разумно. Это заметно упрощает чтение и может ускорить ревью. Обычно это удобнее и для вас: вы получите ранний фидбек и не потратите время на изменения, которые потом придется переделывать.

- **Начинайте каждый PR с короткого, простого описания того, что он делает, а для геймплейных PR прикладывайте ссылки на дизайн-документы, если они есть.** Когда сразу понятно, что делает PR и как он связан с принятыми предложениями из дискорда. Для небольших изменений или когда подходящего предложения в дискорде нет, дизайн можно описать прямо в тексте PR.

## Тестирование

- **Тестируйте все изменения в игре.** Все багфиксы и фичи должны быть проверены в игре на локально запущенной сборке. Также стоит проверить другие фичи, которые могут быть косвенно затронуты.

    - По этой причине **не используйте веб-редактор GitHub для PR.** Веб-правки могут быть закрыты по усмотрению мейнтейнера. Повторная отправка PR, сделанных через веб-редактор, может привести к бану в репозитории.

- **Прикладывайте скриншоты или видео**, показывающие выполненное тестирование. Это также облегчает написание progress-отчетов.

## Перед отправкой

- Определите/подтвердите, является ли ваше изменение срочным **Хотфиксы**.
    - Хотфиксы после мерджа не ждет следующего двухнедельного релиза.
    - Баги, влияющие на Master, и срочные изменения баланса могут считаться Хотфиксами.
    - При открытии PR на GitHub по умолчанию всегда выбрана ветка master.
    - Если не уверены, должен ли change быть Хотфиксы, открывайте PR в master и спросите в обсуждении PR, можно ли оформить его как Хотфиксы.

- Решите, открываете ли вы PR как **draft**.
    - Draft PR допускается только если какая-то часть PR требует review и/или approval для завершения других частей этого же PR.
    - Нельзя открывать draft PR просто потому, что PR еще не готов к ревью.
    - Несоблюдение этого правила может привести к закрытию PR.

- **Проверьте свой diff** через вкладку предпросмотра кода на GitHub.

    - Убедитесь, что нет изменений, которые вы не планировали коммитить.
    - Проверьте случайные пробелы и изменения окончаний строк.

### Заполнение шаблона PR
При отправке pull request на GitHub вы обязаны заполнить актуальный шаблон репозитория.

1. Раздел _Описание PR_ должен кратко и по делу объяснять, что именно изменено.
2. Раздел _Почему / Зачем / Баланс_ должен объяснять, зачем нужны изменения и как они влияют на игру.
   1. Для небольших багфиксов достаточно короткого обоснования.
   2. Для балансных изменений обязательно объясните, **почему** это нужно, а не только что изменилось.
   3. По возможности приложите ссылки на обсуждение, предложение или issue.
3. Раздел _Ссылка на задачу (Если есть)_ заполняйте ссылками на issue/обсуждение/предложение, если они существуют.
4. Раздел _Медиа_ обязателен для изменений, которые видно или проверяются в игре (визуал, механики, контентные фиксы).
   1. Для чисто технических правок и очевидных рефакторингов медиа можно не прикладывать.
5. Раздел _Требования_ должен быть отмечен честно по каждому пункту чеклиста.
   1. PR должен быть завершен и готов к ревью (не используйте draft PR для незавершенной работы).
   2. Вы должны проверить свой diff на случайные изменения.
   3. Изменения должны быть протестированы локально в игре.
6. Раздел _Критические изменения_ заполняйте, если есть breaking changes.
   1. К ним относятся изменения публичных API, переезды/переименования namespace, а также изменения/удаления ID прототипов.
   2. Если изменение нетривиальное, добавьте краткую инструкцию по миграции для других разработчиков.
7. Раздел _Список изменений_ (`:cl:`) заполняйте, когда это важно для игроков.
   1. Используйте формат changelog из шаблона PR.
   2. Для контента RedStar ориентируйтесь на правила из раздела Changelog ниже и актуальные категории.

## После отправки

Вы можете вносить изменения в PR после публикации, например улучшения или исправления багов, найденных уже после отправки.

- **Не делайте force push в ветку** после получения ревью, если мейнтейнер явно не попросил это сделать. Иначе все ревью помечаются как 'outdated', даже если замечания еще не были закрыты.

# Ревью

Ревью - важная часть процесса pull request. Ревью помогают получить фидбек от сообщества и поддерживать высокое качество кода. Так как мейнтейнеры - волонтеры, просим о терпении. Ревью крупных изменений может занять до нескольких месяцев.

Мы ценим и учитываем весь фидбек игроков. Однако финальное решение о том, какие изменения будут влиты в репозиторий, остается за мейнтейнерами. Даже если PR смержен, он может быть откатан на следующей встрече мейнтейнеров и не попасть в следующий stable-релиз. Подробнее смотрите в [модели релизов](https://docs.spacestation14.com/en/general-development/codebase-info/releases.html).

## Как получить ревью

- Ревьюить PR может любой желающий. Ревью от других контрибьюторов может быть не менее ценным, чем ревью от мейнтейнеров; часто это ускоряет мердж и снижает нагрузку на мейнтейнеров. Если вы ждете ревью, полезно найти другого контрибьютора в похожей ситуации и взаимно поревьюить PR друг друга. Чтение чужих PR и критическая оценка того, как бы вы написали код, также отлично прокачивает навыки.

- Мейнтейнеры периодически просматривают открытые PR.

- Если первичное ревью задерживается на несколько дней, нормально попросить ревью в #pr-review-request.

## Как отвечать на ревью

- Когда вы внесли правки по ревью, нажмите 'Resolve conversation' на GitHub после пуша обновленного кода.

- Если у вас есть вопросы по замечаниям в PR (или по коду в целом), смело просите уточнения у ревьюера на GitHub/Discord или в #howdoicode.

# Changelog
Записи changelog помогают игрокам узнавать о новых фичах и изменениях существующих.

## Шаблон changelog
В этом репозитории changelog берется из блока `:cl:` в описании PR и автоматически переносится ботом в файлы changelog RedStar.
Используйте формат из шаблона PR:

```
:cl:
- Добавлено: Добавлено веселье.
- Исправлено: Исправлено веселье.
- Удалено: Удалено веселье.
- Изменено: Изменено веселье.
```

По умолчанию изменения подписываются вашим GitHub-ником. Если хотите, чтобы в игре отображалось другое имя, добавьте строку на той же строке, что и `:cl:`, с нужным именем.

Категория записи должна соответствовать формату шаблона PR (`Добавлено`, `Исправлено`, `Удалено`, `Изменено`). В каждой категории может быть несколько строк.

Мейнтейнеры могут по своему усмотрению добавить, изменить или удалить предложенную вами запись changelog.

### Категории changelog
По умолчанию записи попадают в основной changelog RedStar (`Resources/Changelog/Changelog.yml`).
Для карт используйте категорию `MAPS:`, чтобы запись попала в changelog карт (`Resources/Changelog/Maps.yml`).
Для админских изменений используйте категорию `ADMIN:`.

Эти категории уменьшают шум в общем changelog и помогают игрокам/админам находить нужные изменения.

Важно: оставляйте `:cl:` в итоговом тексте PR (не в HTML-комментарии), иначе бот не обработает запись.

#### Changelog карт
Если указать `MAPS:` в блоке changelog, все записи ниже пойдут в `Resources/Changelog/Maps.yml`.

```
:cl:
MAPS:
- Добавлено: На Meta добавлена новая арена лазертага в северо-восточных техах.
- Удалено: На Bagel удален терминал ID-карт на checkpoint эвакуационной охраны.
- Изменено: На Box ядро ИИ перенесено в центр станции под мостик.
- Исправлено: На Fland переделана проводка массива SMES, теперь он корректно заряжается.
```

Для map changelog желательно явно указывать станцию в каждой строке (`На Box`, `На Meta` и т.д.).
Если изменение затрагивает много станций, используйте формулировки `На многих станциях` или `На всех станциях`.

```
:cl:
MAPS:
- Удалено: На многих станциях удален генератор аномалий.
- Удалено: На всех станциях удален терминал управления грузовым шаттлом.
```

Для добавления/удаления станции допустима более свободная формулировка:
```
:cl:
MAPS:
- Добавлено: Добавлена новая станция RA-12 Spire, инженерно-ориентированная станция для среднего онлайна.
- Добавлено: Добавлена новая станция Gate, станция для высокого онлайна.
- Удалено: Станция Core удалена из ротации.
```

#### Admin changelog

Если указать `ADMIN:`, все записи ниже будут помечены как админские изменения.


```
:cl:
ADMIN:
- Добавлено: Добавлена новая команда для администраторов.
- Исправлено: Исправлено логирование admin-jump.
- Удалено: Удалена устаревшая admin-команда.
- Изменено: Обновлены права для admin-инструмента.
```
или
```
:cl:
- Добавлено: Добавлено изменение для игроков.
ADMIN:
- Изменено: Обновлены админские логи.
```

## Как написать эффективный changelog
Changelog нужен в первую очередь для игроков. Он должен коротко объяснять, что изменилось в игре и как это повлияет на матч. Технические детали, внутренняя кухня и мотивация разработки должны оставаться в описании PR.

При написании changelog придерживайтесь следующих правил:

1. **Пишите понятными и завершенными фразами.** Начинайте с заглавной буквы, заканчивайте точкой.
   - Плохо: `исправлен баг с энергомечом`
   - Хорошо: `Исправлен баг, из-за которого энергомеч иногда не наносил урон.`

2. **Добавляйте только то, что важно игрокам.** В changelog должны попадать изменения геймплея, баланса, доступности контента, важных механик и заметных визуальных апдейтов.
   - Не добавляйте мелкие технические рефакторы, переименования и внутренние правки без игрового эффекта.

3. **Пишите кратко и по сути.** Одна строка changelog = одна мысль.
   - Если изменения однотипные, объединяйте их в одну запись вместо 5-10 почти одинаковых пунктов.

4. **Используйте нейтральный и профессиональный тон.** Без мемов, ролевого отыгрыша и неоднозначных формулировок.
   - Плохо: `ГШ опять урезала бюджет, так что ручки больше не завозят.`
   - Хорошо: `Космические ручки больше недоступны.`

5. **Выбирайте правильную категорию в `:cl:`.**
   - `Добавлено` - новый контент/возможность.
   - `Исправлено` - исправление багов.
   - `Изменено` - баланс/поведение/переработка существующего.
   - `Удалено` - удаление контента или функционала.

6. **Используйте специализированные секции, когда это нужно.**
   - `MAPS:` для изменений карт (с указанием станции в строке, где применимо).
   - `ADMIN:` для админских изменений, неинтересных большинству игроков.
