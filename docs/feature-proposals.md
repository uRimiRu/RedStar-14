# Предложения фич (Feature Proposals)

См. также:
- [Конвенции кода](./conventions.md)
- [Руководство по Pull Request](./pull-request-guidelines.md)
- [Git для разработчика SS14](./git-for-the-ss14-developer.md)


Если вы планируете добавить крупную механику или серьезно переработать существующую систему, сначала оформите proposal, а уже потом начинайте код.

Это экономит время: заранее согласованный дизайн уменьшает риск многократных переделок PR.

## Как оформить proposal

1. Скопируйте шаблон proposal из `src/en/templates/proposal.md` (или актуального каталога шаблонов в вашей ветке документации).
2. Для геймплейных изменений прочитайте [Core Design SS14](https://docs.spacestation14.com/en/space-station-14/core-design.html).
3. Напишите proposal (см. [гайд по редактированию документации](https://docs.spacestation14.com/en/meta/guide-to-editing-docs.html)).
4. Откройте pull request с proposal.
5. Proposal считается принятым после мерджа мейнтейнером.

```admonish tip "Незавершенные proposal"
Если идея еще сырая, но вы хотите ранний фидбек, открывайте PR как draft.
```

## Что такое design doc

Design doc - это описание идеи на уровне дизайна, а не реализации.

Обычно он отвечает на три вопроса:

1. **Зачем** нужна фича.
2. **Что** она добавляет в игру на высоком уровне.
3. **Как** игроки будут с ней взаимодействовать, и как она встраивается в существующие системы.

Вам не нужно расписывать каждый числовой параметр. Важнее логика, роль механики и влияние на игру.

```admonish warning "Чего не делать"
- Не пишите псевдокод и низкоуровневую реализацию.
- Не превращайте proposal в полноценный ТЗ-документ (SRS/FSD).
- Не опускайте взаимодействие игрока с механикой.
- Не фиксируйте детальный баланс на этапе proposal.
```

## Когда proposal обязателен

Как правило, proposal нужен, если вы:
- создаете новую крупную подсистему;
- перепроектируете существующую большую систему;
- меняете целый геймплейный контур (например, профессию/департамент/ключевую механику).

Если изменение масштаба "добавить один предмет/небольшой фикс", чаще всего proposal не требуется.

Практическое правило: если под изменение пришлось бы существенно переписать guidebook или отдельную feature-страницу, скорее всего нужен proposal.

## Примут ли мой proposal

Решение принимают мейнтейнеры через обычный PR-review процесс.

Критика и доработки - нормальная часть процесса. Это не отклонение идеи, а ее шлифовка до состояния, пригодного для внедрения.

```admonish tip "Как повысить шанс принятия"
- Сверьтесь с [Core Design SS14](https://docs.spacestation14.com/en/space-station-14/core-design.html).
- Покажите влияние на текущий геймплей и роль в экосистеме проекта.
- Соблюдайте рабочую этику обсуждений и формат PR.
```

## Для репозитория RedStar

При proposal и последующей реализации учитывайте локальные правила репозитория:
- новый RedStar-контент размещается в `_RedStar`-подкаталогах;
- правки существующих upstream-файлов помечаются `RedStar`-комментариями;
- изменения `RobustToolbox` не вносятся в PR этого репозитория;
- описание и changelog оформляются по вашему шаблону PR.
